# Feature Specification: 저장소 — 폴더/문서 관리 (M3)

**Feature Branch**: `feature/init`

**Created**: 2026-06-26

**Status**: Draft

**Input**: M3 저장소 — IndexedDB 스키마, 폴더 CRUD, `[+]` 버튼, 폴더>문서 트리, 다중 문서 (requirement.md §12 M3, FR-2·§5–6)

## Clarifications

### Session 2026-06-26

- Q: 폴더 중첩(폴더 안 폴더)을 지원하나? → A: M3는 **1단계 폴더 > 문서**만(중첩 없음). 데이터에
  상위 폴더 참조 자리는 미래용으로 예약하되 UI는 평면 폴더 목록.
- Q: 새 문서는 어떻게 만드나? → A: 각 폴더 행의 **"새 문서" 액션**으로 해당 폴더에 빈 문서를 만든다.
- Q: 기존 단일 문서(M1 로컬 저장본)는? → A: 첫 실행 시 **"내 문서" 기본 폴더의 문서 1건으로 이관**한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 폴더를 만들고 문서를 정리 (Priority: P1)

사용자가 사이드바의 `[+]` 버튼으로 폴더를 만들고, 폴더 안에 문서를 생성해 작업을 분류한다.
사이드바에 폴더 > 문서가 트리로 표시된다.

**Why this priority**: M3의 핵심 — 폴더/문서 구조를 도입한다. 이게 없으면 단일 문서에 머문다.

**Independent Test**: `[+]`로 폴더를 만들고 그 폴더에 새 문서를 추가했을 때, 사이드바 트리에
`폴더 > 문서`가 나타나는지로 검증.

**Acceptance Scenarios**:

1. **Given** 사이드바 표시, **When** `[+]` 폴더 생성 후 이름 입력, **Then** 트리에 새 폴더가 추가된다.
2. **Given** 폴더 존재, **When** 그 폴더의 "새 문서" 액션 실행, **Then** 폴더 아래 빈 문서가 생성되고
   트리에 표시된다.
3. **Given** 여러 폴더·문서, **When** 사이드바 확인, **Then** 폴더 > 문서 계층이 트리로 보인다.

---

### User Story 2 - 문서 선택·편집·저장 (다중 문서) (Priority: P1)

사용자가 트리에서 문서를 클릭하면 에디터에 그 문서가 로드되고, 편집 후 저장하면 해당 문서에
내용이 기록된다. 다른 문서를 클릭하면 그 문서로 전환된다.

**Why this priority**: 폴더/문서가 있어도 열고 저장하지 못하면 무의미. 다중 문서 편집이 M3의 가치.

**Independent Test**: 문서 A를 작성·저장 → 문서 B로 전환·작성·저장 → 다시 A 클릭 시 A의 내용이
정확히 복원되는지로 검증.

**Acceptance Scenarios**:

1. **Given** 트리에 문서 2개, **When** 문서 A 클릭, **Then** 에디터에 A 내용이 로드된다.
2. **Given** A 편집 중, **When** 저장, **Then** A 문서에 내용이 기록되고 피드백이 표시된다.
3. **Given** A 저장 후, **When** 문서 B 클릭 → 편집 → 저장 → A 재클릭, **Then** A 내용이 그대로 복원된다.
4. **Given** A에 미저장 변경, **When** 다른 문서 클릭, **Then** 저장되지 않은 변경에 대해 경고/확인한다.

---

### User Story 3 - 폴더·문서 영속 (재방문 보존 + 이관) (Priority: P1)

사용자가 만든 폴더·문서가 새로고침·재방문 후에도 보존된다. 기존 단일 문서(M1 저장본)는 첫 실행 시
기본 폴더의 문서로 자동 이관된다.

**Why this priority**: 데이터가 휘발되면 폴더 구조가 의미 없다. 영속 + 기존 데이터 보존이 신뢰의 기반.

**Independent Test**: 폴더·문서를 만들고 새로고침했을 때 트리와 내용이 그대로인지, 기존 단일 저장본이
"내 문서" 폴더에 나타나는지로 검증.

**Acceptance Scenarios**:

1. **Given** 폴더·문서 작성, **When** 새로고침, **Then** 트리·문서 내용이 100% 보존된다.
2. **Given** M1 단일 저장본이 있는 사용자, **When** 첫 M3 실행, **Then** 그 내용이 "내 문서" 폴더의
   문서 1건으로 이관되어 보인다.

---

### User Story 4 - 폴더/문서 이름변경·삭제 (Priority: P2)

사용자가 폴더·문서의 이름을 바꾸거나 삭제한다.

**Why this priority**: 정리 편의를 높이나, 생성·열기·저장(P1)만으로도 핵심 가치는 성립.

**Independent Test**: 폴더 이름변경 후 트리에 반영되는지, 문서 삭제 후 트리에서 사라지고 영속되는지로 검증.

**Acceptance Scenarios**:

1. **Given** 폴더, **When** 이름변경, **Then** 트리에 새 이름이 반영·영속된다.
2. **Given** 문서, **When** 삭제 확인, **Then** 트리에서 제거되고 재방문 후에도 없다.
3. **Given** 문서가 있는 폴더, **When** 폴더 삭제, **Then** 하위 문서 처리(함께 삭제)를 확인 후 수행한다.

---

### User Story 5 - 폴더 접기/펼치기 (Priority: P3)

사용자가 폴더를 접거나 펼쳐 트리를 정리한다.

**Independent Test**: 폴더를 접으면 하위 문서가 숨고, 펼치면 다시 보이는지로 검증.

**Acceptance Scenarios**:

1. **Given** 펼쳐진 폴더, **When** 폴더 토글, **Then** 하위 문서가 접힌다.

---

### Edge Cases

- 폴더가 하나도 없을 때 새 문서를 만들려 하면? → "내 문서" 기본 폴더를 자동 생성해 거기에 담는다.
- 마지막 문서를 삭제하면? → 에디터는 빈 상태 또는 안내를 보여주고 앱이 깨지지 않는다.
- 같은 폴더에 같은 이름 문서? → 허용하되 식별 가능해야 한다(생성 시각 등으로 구분).
- 활성 문서를 삭제하면? → 다른 문서로 전환하거나 빈 상태로 전환한다.
- 저장 공간 부족·쓰기 실패 시? → 실패를 알리고 데이터 손상이 없어야 한다.
- 이관 중 기존 저장본이 손상돼 있으면? → 빈 기본 문서로 안전하게 시작한다.

## Requirements *(mandatory)*

### Functional Requirements

**폴더/문서 구조**

- **FR-001**: 사용자는 사이드바의 `[+]` 버튼으로 새 폴더를 만들 수 있어야 한다.
- **FR-002**: 사용자는 폴더 안에 새 문서를 만들 수 있어야 한다.
- **FR-003**: 시스템은 사이드바에 폴더 > 문서 계층을 트리로 표시해야 한다(1단계 폴더).
- **FR-004**: 사용자는 폴더를 접거나 펼칠 수 있어야 한다(권장).

**다중 문서 편집**

- **FR-005**: 사용자가 트리에서 문서를 클릭하면 시스템은 그 문서를 에디터에 로드해야 한다.
- **FR-006**: 사용자가 저장하면 시스템은 현재 활성 문서에 내용을 기록해야 한다.
- **FR-007**: 활성 문서에 저장되지 않은 변경이 있는 상태에서 다른 문서로 전환하려 하면 시스템은
  경고/확인해야 한다.

**영속·이관**

- **FR-008**: 시스템은 폴더·문서를 영속 저장하여 새로고침·재방문 후에도 보존해야 한다.
- **FR-009**: 첫 실행 시 시스템은 기존 단일 저장본을 "내 문서" 기본 폴더의 문서 1건으로 이관해야 한다.

**이름변경·삭제**

- **FR-010**: 사용자는 폴더·문서의 이름을 변경할 수 있어야 한다(권장).
- **FR-011**: 사용자는 폴더·문서를 삭제할 수 있어야 하며, 폴더 삭제 시 하위 문서 처리를 확인해야 한다(권장).

**공통·회귀**

- **FR-012**: 폴더/문서 도입이 기존 에디터·프리뷰·툴바·플러그인 동작을 깨지 않아야 한다(회귀 0).
- **FR-013**: 사이드바 트리·액션은 한글 접근성 레이블을 가지며 키보드로 조작 가능해야 한다.

### Key Entities

- **폴더(Folder)**: 문서를 담는 분류 단위. 이름, 생성 시각, (미래용) 상위 폴더 참조. 1 폴더 : N 문서.
- **문서(Document)**: 마크다운 텍스트 1건. 소속 폴더, 제목, 본문, 생성·수정 시각. 폴더에 속한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 폴더·문서를 만들면 사이드바 트리에 폴더 > 문서로 즉시 표시된다.
- **SC-002**: 문서 전환 시 활성 문서가 0.3초 이내에 에디터에 로드된다.
- **SC-003**: 새로고침·재방문 후 폴더·문서·내용이 100% 동일하게 보존된다.
- **SC-004**: 기존 단일 저장본 사용자의 내용이 첫 실행 시 손실 0으로 기본 폴더에 이관된다.
- **SC-005**: 문서 A→B→A 전환 시 각 문서 내용이 서로 섞이지 않고 정확히 복원된다.
- **SC-006**: 기존 M1·M2 기능(에디터·프리뷰·툴바·활성표시·표편집)의 회귀가 0건이다.
- **SC-007**: 사이드바 트리·생성·이름변경·삭제가 키보드로 접근 가능하고 한글 레이블을 노출한다.

## Assumptions

- 폴더는 1단계(폴더 > 문서). 폴더 중첩은 데이터에 참조 자리만 예약하고 UI는 평면 — 후속.
- 새 문서 생성은 폴더별 "새 문서" 액션으로, 빈 본문 문서를 만든다. 문서 제목 기본값은 자동 부여(예: "제목 없음").
- 기존 단일 저장본은 "내 문서" 기본 폴더의 문서 1건으로 이관. 폴더가 없으면 기본 폴더를 자동 생성한다.
- 저장은 현재 활성 문서를 대상으로 한다. "저장 시 폴더 선택"·대시보드·파일 리스트·라우팅은 M6(문서 흐름) 범위.
- 사이드바 리사이즈(M2)는 그대로 계승한다. 시각 토큰·anti-slop 규칙도 M1 확정분 준수.
- 데이터는 사용자 브라우저에만 영속(서버·계정 없음). 모바일은 데스크톱 우선 원칙대로 범위 밖(선택).
