# Feature Specification: 마감 — 성능·접근성·안전종료·배포 검증 (M7)

**Feature Branch**: `007-release-hardening`

**Created**: 2026-06-27

**Status**: Draft

**Input**: User description: "m7 진행"

## 개요

M1–M6으로 기능은 완성됐다(에디터·툴바·저장소·문서흐름). M7은 **신규 기능을 추가하지 않는다.**
대신 그동안 미뤄둔 **비기능 요구(NFR)를 측정 가능한 기준으로 닫고**, 전 마일스톤에 흩어진
수동 검증을 정식 수용 시나리오로 묶어 **출시 가능 상태**를 보증한다. 범위는 requirement.md §12 M7
("보안·접근성·토스트·dirty check(NFR)")와 NFR-1~7, 그리고 005 품질 체크리스트가 지적한 정량화
빈틈(성능 임계값·키보드 리사이즈·구문강조 닫힌목록·대비)이다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 빠른 편집(성능 마감) (Priority: P1)

작성자가 큰 문서(1만 자)를 편집할 때 입력이 끊기거나 프리뷰 갱신이 밀려 답답함을 느끼지 않는다.
타이핑은 즉시 화면에 반영되고, 프리뷰는 짧은 지연 뒤 정해진 한계 안에서 갱신된다.

**Why this priority**: 에디터의 1차 가치는 "쓰는 동안 막힘 없음"이다. 성능 체감이 무너지면
나머지 기능의 완성도와 무관하게 제품이 실패한 것으로 인식된다. 그래서 P1.

**Independent Test**: 1만 자 문서에서 연속 타이핑하며 입력→문자 표시 지연과 입력→프리뷰 갱신
지연을 측정한다. 정해진 임계값(아래 SC) 안이면 통과.

**Acceptance Scenarios**:

1. **Given** 1만 자 분량 문서가 열려 있고, **When** 사용자가 본문 중간에서 연속 입력하면,
   **Then** 입력한 글자는 한 프레임 안에 에디터에 나타나고 입력이 끊기지 않는다.
2. **Given** 같은 문서에서, **When** 입력이 잠시 멎으면, **Then** 프리뷰는 디바운스 한계
   (≤150ms) 안에 최신 내용으로 1회 갱신된다(입력 중 매 글자 재렌더 금지).
3. **Given** 한글 IME 조합 입력 중, **When** 조합이 진행되면, **Then** 조합 중간 상태가
   프리뷰를 깜빡이게 하거나 입력을 가로채지 않는다.

### User Story 2 - 키보드·보조기술 사용자 접근(접근성 마감) (Priority: P1)

마우스를 쓰지 않는 사용자가 키보드만으로 모든 핵심 동작(툴바 서식, 폴더·문서 트리 탐색,
영역 크기 조절, 폴더 선택, 대시보드 이동)에 도달·실행할 수 있고, 스크린리더가 각 컨트롤의
한글 이름을 읽는다.

**Why this priority**: 접근성은 requirement.md NFR-3의 명시 요구이며 출시 게이트다. 키보드
도달 불가 컨트롤이 하나라도 있으면 해당 기능은 일부 사용자에게 존재하지 않는 것과 같다. P1.

**Independent Test**: 마우스를 쓰지 않고 Tab/Shift+Tab/Enter/방향키만으로 서식 적용·문서 전환·
영역 리사이즈·폴더 선택·대시보드↔에디터 이동을 끝까지 수행한다. 모든 인터랙티브 요소에
한글 접근성 이름이 있는지, 색 대비가 기준(WCAG AA)을 만족하는지 점검한다.

**Acceptance Scenarios**:

1. **Given** 에디터 화면에서, **When** 키보드 포커스를 순회하면, **Then** 모든 툴바 버튼·트리
   항목·리사이즈 핸들·폴더 선택·대시보드 링크가 포커스를 받고 보이는 포커스 표시가 나타난다.
2. **Given** 영역 경계(에디터:프리뷰, 사이드바:컨텐츠)에 포커스가 있을 때, **When** 방향키를
   누르면, **Then** 경계가 정해진 증분만큼 이동하고 최소·최대 한계에서 멈춘다.
3. **Given** 스크린리더가 켜진 상태에서, **When** 각 컨트롤에 포커스가 가면, **Then** 한글
   이름(예: "굵게", "새 문서", "영역 크기 조절")이 읽힌다.
4. **Given** 프리뷰·에디터 구문 강조·상태 텍스트에서, **When** 색 대비를 측정하면, **Then**
   본문 텍스트는 WCAG AA(4.5:1) 이상을 만족한다.

### User Story 3 - 안전한 종료와 데이터 보호 (Priority: P1)

사용자가 저장하지 않은 변경을 잃지 않고, 원시 HTML을 입력해도 프리뷰가 안전하며, 저장·실패가
명확한 피드백으로 보인다.

**Why this priority**: 데이터 유실과 XSS는 신뢰를 즉시 무너뜨린다. sanitize(NFR-1)는 헌법 II
NON-NEGOTIABLE, dirty check·영속(NFR-7)은 데이터 보호의 핵심이라 P1.

**Independent Test**: ① 변경 후 미저장 상태에서 문서 전환·새로고침·대시보드 이동 시 경고가
뜨는지, ② 악의적 HTML(`<script>`, `onerror=` 등)을 입력해도 프리뷰에서 실행되지 않는지,
③ 저장 성공·실패가 토스트로 구분돼 보이는지 확인.

**Acceptance Scenarios**:

1. **Given** 저장하지 않은 변경이 있을 때, **When** 다른 문서로 이동/새로고침/대시보드 이동을
   시도하면, **Then** 변경을 버릴지 묻는 경고가 뜨고, 취소 시 현재 문서·내용이 유지된다.
2. **Given** 본문에 `<script>`·`<img onerror=...>` 같은 원시 HTML이 있을 때, **When** 프리뷰가
   렌더되면, **Then** 스크립트·이벤트 핸들러는 제거되어 실행되지 않는다.
3. **Given** 저장을 시도할 때, **When** 저장이 성공하면 성공 토스트가, 실패하면 실패 토스트가
   뜨고, **Then** 두 경우가 시각적으로 구분된다.
4. **Given** 저장 후 새로고침·재방문하면, **Then** 폴더·문서·내용이 그대로 보존된다.

---

### User Story 4 - 무설정 배포 (Priority: P2)

운영자가 별도 서버 설정 없이 정적 호스팅(Vercel)에 빌드·배포하면 앱이 그대로 동작한다.

**Why this priority**: 배포 가능성은 출시의 전제지만, P1 품질 게이트가 충족된 뒤 검증해도
늦지 않아 P2.

**Independent Test**: 클린 환경에서 프로덕션 빌드를 수행하고, 서버 전용 기능 없이 정적+클라이언트로
모든 핵심 흐름(편집·저장·문서 전환·새로고침 영속)이 동작하는지 확인.

**Acceptance Scenarios**:

1. **Given** 클린 체크아웃에서, **When** 프로덕션 빌드를 수행하면, **Then** 추가 설정 없이
   빌드가 성공한다.
2. **Given** 배포본에서, **When** 네트워크 서버 응답 없이 앱을 사용하면, **Then** 편집·저장·문서
   전환·영속이 모두 브라우저 안에서 동작한다.

---

### Edge Cases

- 매우 짧은 문서(스크롤 불가)에서 스크롤 동기화·성능 측정이 의미 있게 동작하는가(되튐·오류 없음)?
- 창이 매우 좁을 때 키보드 리사이즈가 양쪽 최소 너비를 침범하지 않는가?
- 빈 문서·빈 폴더·문서 0개 상태에서 접근성 이름·포커스 순서가 깨지지 않는가?
- 저장 실패(스토리지 용량 초과·접근 거부) 시 데이터가 유실되지 않고 실패 토스트가 뜨는가?
- 매우 긴 한 줄(줄바꿈 없는 1만 자)에서 입력 지연·구문 강조가 임계값을 넘지 않는가?

## Requirements *(mandatory)*

### Functional Requirements

**성능 (NFR-2)**

- **FR-001**: 시스템은 입력 중 매 글자마다 프리뷰를 재렌더하지 않고, 입력이 멎은 뒤 **150ms 이하**
  디바운스 후 1회 갱신해야 한다.
- **FR-002**: 1만 자 문서에서 연속 입력 시, 입력한 글자는 끊김 없이 에디터에 즉시 표시되어야 한다
  (입력 지연 체감 없음).
- **FR-003**: 한글 IME 조합 입력은 조합 중간 상태로 프리뷰를 갱신하거나 입력을 가로채지 않아야
  한다.

**접근성 (NFR-3)**

- **FR-004**: 모든 인터랙티브 요소(툴바 버튼·폴더/문서 트리 항목·리사이즈 핸들·폴더 선택·대시보드
  링크·토스트 닫기)는 키보드로 포커스·실행 가능해야 하고, 보이는 포커스 표시를 가져야 한다.
- **FR-005**: 모든 인터랙티브 요소는 **한글 접근성 이름**(aria-label 또는 가시 텍스트)을 가져야
  한다.
- **FR-006**: 영역 경계(에디터:프리뷰, 사이드바:컨텐츠)는 키보드 방향키로 **정해진 증분(화살표
  ±5%p, Home/End 최소·최대)** 만큼 조절 가능하고, 최소·최대 한계(사이드바 180–480px, 영역 25%)에서
  멈춰야 한다.
- **FR-007**: 본문 텍스트·구문 강조·상태 텍스트의 색 대비는 **WCAG AA(일반 텍스트 4.5:1)** 이상을
  만족해야 한다.

**안전 종료·데이터 (NFR-1, NFR-7)**

- **FR-008**: 저장하지 않은 변경(dirty)이 있을 때 문서 전환·새로고침·대시보드 이동을 시도하면
  변경 손실 경고를 표시하고, 취소 시 현재 상태를 유지해야 한다.
- **FR-009**: 프리뷰 렌더는 원시 HTML의 스크립트·이벤트 핸들러·위험 속성을 제거해 실행을 차단해야
  한다(sanitize 항상 적용).
- **FR-010**: 저장 성공과 실패는 각각 시각적으로 구분되는 토스트로 사용자에게 알려야 한다.
- **FR-011**: 저장된 폴더·문서·내용은 새로고침·재방문 후에도 보존되어야 한다.

**배포 (NFR-6)**

- **FR-012**: 프로덕션 빌드는 추가 서버 설정 없이 성공해야 하며, 앱은 서버 전용 기능 없이
  정적+클라이언트만으로 핵심 흐름을 제공해야 한다.

**회귀**

- **FR-013**: 본 마감 작업은 M1–M6의 기존 기능(렌더·툴바·활성표시·저장·다중문서·라우팅·폴더선택)을
  깨지 않아야 하며, 기존 자동화 테스트는 전부 통과(GREEN)를 유지해야 한다.

### Key Entities

해당 없음 — 신규 데이터 모델 없음. 기존 `folders`/`documents` 영속 모델을 재사용한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 1만 자 문서에서 입력→문자 표시 지연이 **한 프레임(약 16ms)** 수준으로, 사용자가
  끊김을 느끼지 않는다.
- **SC-002**: 입력이 멎은 뒤 프리뷰 갱신은 **150ms 이하**에 1회 발생한다(입력 중 글자당 재렌더 0회).
- **SC-003**: 키보드만으로 서식 적용·문서 전환·영역 리사이즈·폴더 선택·대시보드 이동을 **마우스
  없이 100% 완수**할 수 있다.
- **SC-004**: 모든 인터랙티브 요소에 한글 접근성 이름이 존재한다(누락 0건).
- **SC-005**: 본문 텍스트 색 대비가 WCAG AA(4.5:1) 이상이다(위반 0건).
- **SC-006**: 미저장 변경 상태에서의 이탈 시도 100%에서 손실 경고가 표시되고, 취소 시 유실 0건.
- **SC-007**: 알려진 XSS 페이로드(스크립트 주입·이벤트 핸들러·위험 속성)가 프리뷰에서 실행되는
  건수 0건.
- **SC-008**: 클린 환경 프로덕션 빌드가 추가 설정 없이 성공하고, 저장·새로고침 영속이 배포본에서
  동작한다.
- **SC-009**: 기존 자동화 테스트가 전부 통과(회귀 0)를 유지한다.

## Assumptions

- **성능 기준값**: requirement.md NFR-2(디바운스 ≤150ms, 1만 자 무체감)를 정량 기준으로 채택한다.
  "1만 자"는 성능 측정의 기준 문서 크기다.
- **접근성 기준**: 색 대비는 WCAG 2.1 AA(일반 텍스트 4.5:1)를 기준으로 한다. 모바일 보조기술은
  범위 밖(NFR-4 데스크톱 우선).
- **키보드 리사이즈**: 영역 경계의 키보드 조작은 기존 리사이즈 라이브러리의 기본 키보드 지원에
  의존하되, 증분·한계 동작이 요구를 충족하는지 검증한다(미흡 시 보강).
- **E2E 검증 방식**: 무거운 브라우저 자동화 프레임워크를 신규 도입하지 않는다. 전 마일스톤의
  수동 검증 시나리오(quickstart의 시각 체크)와 기존 Vitest 단위/통합을 정식 수용 기준으로 묶는다.
- **sanitize·영속**: 헌법 II sanitize 파이프라인과 IndexedDB 영속(M3/M6)을 재사용한다. 본 작업은
  새 저장 모델·새 렌더 경로를 만들지 않고 **검증·정량화·접근성 보강**에 한정한다.
- **모바일·국제화·코드블록 언어 하이라이트·수식(KaTeX)**: 범위 밖(선택 기능, requirement.md NFR-4·
  005 Assumptions와 일치).
