**Feature Branch**: `feature/init`

**Created**: 2026-06-26

**Status**: Draft

**Input**: M2 레이아웃 — 좌측 사이드바 + 컨텐츠 2분할 + 리사이즈 (requirement.md §12 M2, FR-1·FR-2.4)

## Clarifications

### Session 2026-06-26

- Q: M2 범위에 폴더/문서 CRUD·데이터 영속을 포함하나? → A: 아니다. requirement.md §12가 M2=레이아웃,
  M3=저장소(IndexedDB·폴더 CRUD·`[+]` 버튼)로 가른다. M2는 사이드바 **레이아웃 셸 + 리사이즈**까지.
  폴더/문서 데이터·CRUD·다중 문서 로드는 M3.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 사이드바가 있는 3분할 레이아웃 (Priority: P1)

사용자가 앱에 진입하면 좌측에 사이드바, 우측에 에디터:프리뷰 컨텐츠가 배치된 레이아웃을 본다.
와이어프레임(상단바 / 사이드바 | (툴바 · 에디터:프리뷰) / 상태바)과 시각적으로 일치한다.

**Why this priority**: M2의 본질 — 사이드바 영역을 화면에 도입한다. 이게 없으면 후속(M3 폴더 트리)이
들어갈 자리가 없다.

**Independent Test**: 앱 진입 시 좌측 사이드바 영역이 렌더되고, 상단바·상태바는 전체 폭, 사이드바
오른쪽에 툴바+에디터+프리뷰가 배치되는지로 검증.

**Acceptance Scenarios**:

1. **Given** 앱 진입, **When** 화면 렌더, **Then** 좌측에 사이드바 패널이 보이고 우측에 기존
   툴바·에디터·프리뷰가 그대로 동작한다.
2. **Given** 사이드바 표시, **When** 사이드바 헤더 확인, **Then** `[+] 폴더 생성` 자리와 트리
   영역(현재는 빈 상태 안내)이 보인다.

---

### User Story 2 - 사이드바 너비 리사이즈 (Priority: P1)

사용자가 사이드바와 컨텐츠 경계를 드래그해 사이드바 너비를 조절한다. 일정 범위 안에서만 조절되어
컨텐츠가 과도하게 좁아지지 않는다.

**Why this priority**: requirement.md FR-2.4 필수. 사이드바 폭은 사용자마다 선호가 달라 조절 가능해야
한다.

**Independent Test**: 경계 핸들을 드래그해 사이드바가 넓어지고/좁아지며, 최소·최대 한계에서 멈추는지로
검증.

**Acceptance Scenarios**:

1. **Given** 기본 너비 사이드바, **When** 경계를 오른쪽으로 드래그, **Then** 사이드바가 넓어진다.
2. **Given** 사이드바 최소 너비, **When** 더 좁히려 드래그, **Then** 최소 한계에서 멈춘다.
3. **Given** 사이드바 최대 너비, **When** 더 넓히려 드래그, **Then** 최대 한계에서 멈춘다.

---

### Edge Cases

- 창이 매우 좁아지면? → 컨텐츠 영역이 최소 폭을 유지하도록 사이드바 최대 비율을 제한한다.
- 사이드바가 비어 있을 때(M2)? → "폴더/문서 관리는 다음 단계에서" 빈 상태 안내를 보여준다.
- 리사이즈 중 에디터 입력/프리뷰 렌더가 깨지지 않아야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 좌측 사이드바 + 우측 컨텐츠(툴바·에디터·프리뷰)의 2분할 레이아웃을 제공해야 한다.
- **FR-002**: 상단바와 상태바는 전체 폭을 유지하고, 사이드바는 그 사이 영역의 좌측을 차지해야 한다.
- **FR-003**: 사용자는 사이드바와 컨텐츠 경계를 드래그해 사이드바 너비를 조절할 수 있어야 한다.
- **FR-004**: 사이드바 너비는 최소·최대 한계 안에서만 조절되어 컨텐츠가 과도하게 좁아지지 않아야 한다.
- **FR-005**: 사이드바는 `[+] 폴더 생성` 자리와 폴더/문서 트리 영역을 와이어프레임 구조로 표시해야
  한다. 데이터·CRUD는 M3 범위이므로 M2에서는 빈 상태 안내를 보여준다.
- **FR-006**: 리사이즈·사이드바 도입이 기존 에디터·프리뷰·툴바·저장 동작을 깨지 않아야 한다(회귀 0).
- **FR-007**: 사이드바·핸들은 한글 접근성 레이블을 가지며 키보드로 조작 가능해야 한다.

### Key Entities

- 없음(M2는 레이아웃 전용). 폴더/문서 엔티티는 M3에서 도입.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 앱 진입 시 사이드바 영역이 항상 렌더되고 와이어프레임 구조(상단바/사이드바|컨텐츠/상태바)와
  시각적으로 일치한다.
- **SC-002**: 사이드바 너비가 드래그로 조절되며 최소·최대 한계에서 멈춘다(컨텐츠 최소 폭 보존).
- **SC-003**: 기존 M1·M5 기능(에디터 입력·실시간 프리뷰·툴바·저장·활성표시) 회귀가 0건이다.
- **SC-004**: 사이드바·리사이즈 핸들이 키보드 접근 가능하고 한글 레이블을 노출한다.

## Assumptions

- 폴더/문서 데이터·CRUD·다중 문서·IndexedDB는 M3. M2는 레이아웃 셸 + 리사이즈만.
- 사이드바 너비 한계는 데스크톱 기준 비율로 근사한다(기본 ~260px, ~180–480px 범위). 픽셀 정밀
  클램프는 후속 미세조정.
- 시각 토큰·anti-slop 규칙은 M1 확정분을 그대로 따른다(신규 사이드바도 동일 비주얼).
- 모바일은 데스크톱 우선 원칙대로 본 마일스톤 범위 밖(선택).
