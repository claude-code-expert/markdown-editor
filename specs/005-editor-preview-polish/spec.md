## User Scenarios & Testing *(mandatory)*

### User Story 1 - 에디터:프리뷰 비율 조절 (Priority: P1)

작성자가 에디터와 프리뷰 사이 경계를 드래그해 두 영역의 너비 비율을 조절한다. 긴 코드를 볼 땐
에디터를, 결과를 확인할 땐 프리뷰를 넓힐 수 있다.

**Why this priority**: 50:50 고정은 상황에 안 맞을 때가 많다. 비율 조절은 작성 효율을 직접 높이는
핵심 보강(PRD A-6).

**Independent Test**: 경계를 좌/우로 드래그해 에디터·프리뷰 너비가 바뀌고, 최소 한계에서 멈추며,
입력·렌더가 정상 동작하는지로 검증.

**Acceptance Scenarios**:

1. **Given** 50:50 분할, **When** 경계를 오른쪽으로 드래그, **Then** 에디터가 넓어지고 프리뷰가 좁아진다.
2. **Given** 한쪽이 최소 너비, **When** 더 좁히려 드래그, **Then** 최소 한계에서 멈춘다(양쪽 모두 가독 폭 보존).
3. **Given** 비율 조절 후, **When** 에디터 입력, **Then** 프리뷰가 정상적으로 실시간 갱신된다.

---

### User Story 2 - 스크롤 동기화 (Priority: P1)

작성자가 에디터를 스크롤하면 프리뷰가 비례해서 따라 움직이고, 반대도 동일하다. 긴 문서에서 보고
있는 위치가 양쪽에서 어긋나지 않는다.

**Why this priority**: 긴 문서에서 에디터 위치와 프리뷰 위치가 따로 놀면 대조가 어렵다. 동기화는
작성-검토 왕복을 매끄럽게 한다(PRD A-7).

**Independent Test**: 긴 문서에서 에디터를 아래로 스크롤하면 프리뷰도 비례해 내려가고, 프리뷰를
스크롤하면 에디터도 따라오는지로 검증.

**Acceptance Scenarios**:

1. **Given** 스크롤이 생기는 긴 문서, **When** 에디터를 50% 지점까지 스크롤, **Then** 프리뷰도 약 50%
   지점으로 이동한다.
2. **Given** 동기화 상태, **When** 프리뷰를 스크롤, **Then** 에디터가 비례해 따라온다.
3. **Given** 빠른 연속 스크롤, **When** 한쪽을 스크롤, **Then** 무한 반동(되튐) 없이 안정적으로 동기화된다.

---

### User Story 3 - 에디터 마크다운 구문 강조 (Priority: P2)

작성자가 에디터에 입력한 마크다운 문법(제목·굵게·기울임·코드·링크·목록 등)이 시각적으로 강조되어,
원시 텍스트 상태에서도 구조를 한눈에 파악한다.

**Why this priority**: 구문 강조는 가독성·작성 속도를 높이나, 핵심 흐름(리사이즈·동기화)을 차단하지
않으므로 P2.

**Independent Test**: 제목·굵게·코드 등을 입력했을 때 에디터에서 해당 문법이 평문과 구분되는 시각
스타일로 표시되는지로 검증.

**Acceptance Scenarios**:

1. **Given** 에디터, **When** `# 제목`·`**굵게**`·`` `코드` `` 입력, **Then** 각 문법이 평문과 다른
   시각 스타일로 강조된다.
2. **Given** 구문 강조 상태, **When** 한글·긴 문서 입력, **Then** 강조가 입력 지연을 체감시키지 않는다.

---

### Edge Cases

- 창이 매우 좁아지면? → 에디터·프리뷰 각각 최소 너비를 유지해 한쪽이 사라지지 않는다.
- 한쪽 문서가 짧아 스크롤이 없으면? → 동기화는 스크롤 가능한 쪽만 따르고 오류 없이 무시한다.
- 스크롤 동기화 중 입력이 일어나면? → 렌더 갱신과 동기화가 충돌해 깜빡이거나 되튀지 않는다.
- 리사이즈/동기화 도입이 기존 실시간 렌더·툴바·저장·다중문서 전환을 깨지 않아야 한다.

## Requirements *(mandatory)*

### Functional Requirements

**경계 리사이즈**

- **FR-001**: 사용자는 에디터와 프리뷰 사이 경계를 드래그해 두 영역의 너비 비율을 조절할 수 있어야 한다.
- **FR-002**: 에디터·프리뷰는 각각 최소 너비 한계를 가져 한쪽이 과도하게 좁아지지 않아야 한다.
- **FR-003**: 비율 조절 후에도 실시간 렌더·입력이 정상 동작해야 한다.

**스크롤 동기화**

- **FR-004**: 한쪽(에디터 또는 프리뷰)을 스크롤하면 다른 쪽이 비례 위치로 이동해야 한다(양방향).
- **FR-005**: 동기화는 무한 반동(피드백 루프) 없이 안정적이어야 한다.
- **FR-006**: 스크롤이 불가능한(짧은) 영역에 대해서는 오류 없이 동기화를 건너뛰어야 한다.

**에디터 구문 강조**

- **FR-007**: 에디터는 입력된 마크다운 문법(제목·강조·코드·링크·목록 등)을 평문과 구분되는 시각
  스타일로 강조해야 한다.
- **FR-008**: 구문 강조는 한글 입력·대용량 문서에서 입력 지연을 체감시키지 않아야 한다.

**공통·회귀**

- **FR-009**: 본 보강이 기존 실시간 렌더·툴바·활성표시·저장·다중문서 전환을 깨지 않아야 한다(회귀 0).
- **FR-010**: 리사이즈 핸들은 한글 접근성 레이블을 가지며 키보드로 조작 가능해야 한다.

### Key Entities

- 없음(렌더·상호작용 보강 전용, 새 데이터 없음).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 경계를 드래그하면 에디터:프리뷰 비율이 즉시 바뀌고 최소 한계에서 멈춘다.
- **SC-002**: 한쪽을 특정 비율 지점까지 스크롤하면 다른 쪽이 비슷한 비율 지점(±10% 이내)으로 이동한다.
- **SC-003**: 스크롤 동기화에서 되튐·깜빡임이 관측되지 않는다.
- **SC-004**: 에디터에서 제목·강조·코드 등 주요 문법이 평문과 구분되어 표시된다.
- **SC-005**: 1만 자 문서에서 리사이즈·동기화·구문 강조가 입력 지연을 체감시키지 않는다.
- **SC-006**: 기존 기능(실시간 렌더·툴바·저장·다중문서) 회귀가 0건이다.
- **SC-007**: 리사이즈 핸들이 키보드로 접근 가능하고 한글 레이블을 노출한다.

## Assumptions

- 에디터:프리뷰 분할은 사이드바와 동일한 리사이즈 메커니즘을 재사용한다. 기본 50:50, 좌우 최소 한계.
- 스크롤 동기화는 보이는 영역의 스크롤 비율(0~1)을 맞추는 비례 방식을 기본으로 한다. 정밀한 소스맵
  기반 줄 단위 매핑은 본 마일스톤 범위 밖(선택).
- 에디터 구문 강조는 마크다운 문법 강조를 도입하되, 코드 블록 내부 언어별 하이라이트·수식(KaTeX)은
  선택 사항으로 본 마일스톤 필수가 아니다.
- 비율·동기화 상태의 영속(새로고침 후 유지)은 선택이며 본 마일스톤 필수가 아니다.
- 시각 토큰·anti-slop 규칙은 기존 확정분을 그대로 따른다. 데스크톱 우선, 모바일은 범위 밖(선택).
