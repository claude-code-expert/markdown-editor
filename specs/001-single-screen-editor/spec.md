## Clarifications

### Session 2026-06-26

- Q: 저장 이력 없는 첫 방문 시 에디터 초기 콘텐츠는? → A: 간결한 온보딩 샘플 마크다운을 시드한다(제목 + 표·체크박스 등 GFM 시연, 사용자가 자유 편집·삭제 가능). 프로토타입의 긴 `SAMPLE_MD` 전문은 사용하지 않는다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 실시간 프리뷰로 마크다운 작성 (Priority: P1)

문서 작성자가 단일 화면에 진입해 좌측 에디터에 마크다운을 입력하면, 우측 프리뷰가
즉시 렌더된 결과를 보여준다. 작성자는 결과물의 최종 모습을 보면서 글을 다듬는다.

**Why this priority**: 제품의 핵심 가치(입력 즉시 정확한 CommonMark+GFM 프리뷰)를 단독으로
구현한다. 이것 하나만 있어도 "프리뷰 가능한 마크다운 뷰어"라는 최소 가치가 성립한다.

**Independent Test**: 에디터에 표·취소선·체크박스·제목·코드블록을 포함한 마크다운을 붙여넣고,
프리뷰가 해당 문법을 정확히 렌더하는지(공식 CommonMark+GFM 출력과 일치) 확인하면 단독 검증된다.

**Acceptance Scenarios**:

1. **Given** 빈 에디터, **When** `# 제목`을 입력, **Then** 프리뷰에 제목 수준 1로 렌더된다.
2. **Given** 에디터에 텍스트 입력 중, **When** 입력을 멈춤, **Then** 0.15초 이내에 프리뷰가 갱신된다.
3. **Given** GFM 표 문법 입력, **When** 프리뷰 렌더, **Then** 표가 행·열 구조로 표시된다.
4. **Given** 사용자가 원시 HTML(`<script>` 등)을 입력, **When** 프리뷰 렌더, **Then** 스크립트는
   실행되지 않고 안전하게 제거·무력화된 상태로 표시된다.

---

### User Story 2 - 툴바로 문법 암기 없이 서식 적용 (Priority: P1)

작성자가 문법을 외우지 않고 툴바 버튼을 눌러 굵게·제목·링크·목록·표 등 서식을 삽입한다.
선택 영역이 있으면 그 영역에, 없으면 커서 위치에 문법이 적용된다.

**Why this priority**: §1 핵심 가치 중 "툴바로 문법 암기 없이 서식 적용"을 담당. 에디터 입력
경험을 완성하는 필수 축이며 P1 프리뷰와 함께 MVP를 구성한다.

**Independent Test**: 텍스트를 선택하고 굵게 버튼을 눌러 `**텍스트**`가 감싸지는지, 다시 누르면
해제되는지, 링크 버튼이 URL 입력을 받아 `[텍스트](url)`을 삽입하는지로 단독 검증된다.

**Acceptance Scenarios**:

1. **Given** "안녕"을 선택, **When** 굵게 버튼 클릭, **Then** "**안녕**"으로 바뀐다.
2. **Given** 이미 굵은 "**안녕**" 선택, **When** 굵게 버튼 재클릭, **Then** "안녕"으로 해제된다.
3. **Given** 커서가 빈 줄, **When** 제목 H2 버튼 클릭, **Then** 줄 앞에 `## ` 접두가 붙는다.
4. **Given** 텍스트 선택, **When** 링크 버튼 클릭 후 URL 입력, **Then** `[텍스트](입력URL)`이 삽입된다.
5. **Given** 키보드 단축키(굵게), **When** 단축키 입력, **Then** 버튼 클릭과 동일하게 동작한다.

---

### User Story 3 - 로컬 저장으로 작업 보존 (Priority: P1)

작성자가 저장하면 문서가 브라우저 로컬에 영속되고, 새로고침·재방문 후에도 마지막 저장본이
자동 복원된다. 계정·서버 없이 동작한다.

**Why this priority**: §1 핵심 가치 "로컬 영속 — 새로고침·재방문 후에도 문서 보존"을 담당.
저장이 없으면 작성 결과가 휘발되어 도구로 성립하지 않으므로 P1.

**Independent Test**: 문서를 작성·저장하고 페이지를 새로고침했을 때 내용이 그대로 복원되는지로
단독 검증된다.

**Acceptance Scenarios**:

1. **Given** 작성된 문서, **When** 저장 버튼 클릭, **Then** "저장되었습니다" 피드백이 표시된다.
2. **Given** 저장 후 페이지 새로고침, **When** 재진입, **Then** 마지막 저장 내용이 자동 로드된다.
3. **Given** 저장 이후 내용을 수정, **When** 취소 버튼 클릭, **Then** 마지막 저장 상태로 복원된다.
4. **Given** 미저장 변경이 있는 상태, **When** 페이지 이탈 시도, **Then** 저장되지 않은 변경 경고가 표시된다.

---

### User Story 4 - 작성 분량 확인 (Priority: P3)

작성자가 하단 상태바에서 현재 문서의 문자수·줄수를 확인한다.

**Why this priority**: 보조 정보 제공으로 가치는 있으나 핵심 흐름(작성·서식·저장)에 차단 요소가
아니다. 권장 수준이라 P3.

**Independent Test**: 텍스트 입력에 따라 상태바의 문자수·줄수가 실시간으로 갱신되는지로 검증.

**Acceptance Scenarios**:

1. **Given** 에디터에 텍스트 입력, **When** 글자 추가/삭제, **Then** 상태바 문자수·줄수가 즉시 갱신된다.

---

### Edge Cases

- 빈 문서를 저장하면? → 빈 내용으로 정상 저장되고 새로고침 후 빈 에디터로 복원된다.
- 매우 큰 문서(1만 자)를 입력하면? → 입력 지연이 체감되지 않고 프리뷰가 디바운스 내에 갱신된다.
- 저장 영역이 가득 차거나 쓰기에 실패하면? → 실패 피드백이 표시되고 에디터 내용은 유지된다.
- 선택 없이 인라인 서식 버튼을 누르면? → 커서 위치에 빈 서식 마커가 삽입되고 커서가 그 안에 놓인다.
- 깨진/저장 불가능한 로컬 데이터가 있으면? → 온보딩 샘플 시드로 안전하게 시작하고 앱이 깨지지 않는다.
- 첫 방문(저장 이력 없음)에서 사용자가 시드 샘플을 지우고 저장하면? → 빈 문서로 저장되며 다음
  방문부터는 시드 대신 빈 저장본이 로드된다(시드는 최초 1회만).
- 한글 입력(IME 조합) 중 프리뷰 갱신이 조합을 방해하지 않아야 한다.

## Requirements *(mandatory)*

### Functional Requirements

**에디터 / 프리뷰**

- **FR-001**: 시스템은 단일 화면에 에디터(좌)와 프리뷰(우)를 50:50 비율로 표시해야 한다.
- **FR-002**: 사용자가 에디터에 입력하면 시스템은 프리뷰를 실시간 갱신해야 한다. 입력이 멈춘
  뒤 갱신을 **시작**하기까지의 대기(디바운스)는 0.15초(150ms) 이하여야 하며, 일반 분량 문서에서
  사용자가 갱신을 "즉시"로 체감해야 한다.
- **FR-003**: 시스템은 CommonMark 0.31.2 코어 문법과 GFM 확장(표·취소선·작업목록·자동링크
  리터럴)을 렌더해야 한다.
- **FR-004**: 시스템은 프리뷰 렌더 전에 사용자가 입력한 원시 HTML을 정제하여, 스크립트 실행 등
  XSS를 차단해야 한다. (정제는 생략 불가)

**툴바**

- **FR-005**: 시스템은 마크다운 서식 버튼(제목 H1–H6, 굵게, 기울임, 취소선, 인라인 코드,
  코드블록, 링크, 이미지, 인용, 순서 없는 목록, 순서 있는 목록, 작업목록, 표, 구분선)을
  툴바에 표시해야 한다.
- **FR-006**: 사용자가 서식 버튼을 클릭하면 시스템은 현재 선택 영역(없으면 커서 위치)에 해당
  마크다운 문법을 삽입해야 한다.
- **FR-007**: 인라인 서식(굵게·기울임·취소선·코드)이 이미 적용된 선택에 같은 버튼을 누르면
  시스템은 해당 서식을 해제(토글)해야 한다.
- **FR-008**: 링크·이미지 버튼은 사용자에게 URL 입력을 받는 다이얼로그를 제공해야 한다.
- **FR-009**: 시스템은 주요 인라인 서식(굵게·기울임 등)에 키보드 단축키를 제공해야 하며,
  단축키 동작은 버튼 클릭과 동일해야 한다.
- **FR-010**: 모든 툴바 버튼은 한글 접근성 레이블을 가지며 키보드로 포커스 가능해야 한다.

**저장 / 로컬 영속**

- **FR-011**: 사용자가 저장하면 시스템은 에디터 내용을 브라우저 로컬에 영속화해야 한다.
- **FR-012**: 시스템은 저장 성공·실패에 대한 사용자 피드백(토스트)을 표시해야 한다.
- **FR-013**: 사용자가 재방문·새로고침하면 시스템은 마지막 저장 내용을 자동 로드해야 한다.
- **FR-014**: 사용자가 취소하면 시스템은 에디터를 마지막 저장 상태로 복원해야 한다.
- **FR-015**: 저장되지 않은 변경이 있는 상태에서 페이지를 이탈하려 하면 시스템은 경고해야 한다.

**상태 표시**

- **FR-016**: 시스템은 하단 상태바에 현재 문서의 문자수와 줄수를 표시하고 입력에 따라 갱신해야 한다.

**범위 경계**

- **FR-017**: 시스템은 단일 문서 1개만 다룬다. 폴더·문서 트리·다중 문서·대시보드는 본 마일스톤
  범위에서 제외한다.
- **FR-018**: 저장 이력이 없는 최초 진입 시 시스템은 간결한 온보딩 샘플 마크다운(제목 +
  표·작업목록 등 GFM 시연 포함)을 에디터에 시드해야 하며, 사용자는 이를 자유롭게 편집·삭제할
  수 있어야 한다. 저장 이력이 존재하면 시드 대신 마지막 저장본을 로드한다(FR-013 우선).

### Key Entities

- **문서(Document)**: 사용자가 작성하는 마크다운 텍스트 1건. 본문 내용과 "마지막 저장본"
  상태를 가진다. M1에서는 단일 문서이며 폴더에 속하지 않는다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 입력을 멈춘 뒤 갱신을 시작하기까지의 대기가 0.15초 이하이고, 일반 분량
  문서에서 프리뷰 갱신이 즉시로 체감된다(대기 + 렌더 합산 지각 지연 없음).
- **SC-002**: 1만 자 분량 문서에서 입력 시 체감 지연 없이 타이핑과 프리뷰 갱신이 동작한다.
- **SC-003**: 마크다운 기능 명세의 **코어(CommonMark) + GFM 확장**(표·취소선·작업목록·자동링크
  리터럴) 적합성(conformance) 검증 항목을 100% 통과한다.
- **SC-004**: 저장 후 새로고침·재방문 시 마지막 저장 내용이 100% 동일하게 복원된다.
- **SC-005**: 사용자가 원시 HTML로 시도한 스크립트가 프리뷰에서 단 1건도 실행되지 않는다.
- **SC-006**: 처음 보는 사용자가 안내 없이 툴바만으로 굵게·제목·링크·목록·표 서식을 적용할 수 있다.
  (사용성 결과 지표 — 자동 테스트 대신 quickstart 검증 시 휴리스틱 점검: 각 버튼이 식별 가능한
  아이콘 + 한글 레이블을 노출하고 클릭 즉시 해당 문법이 삽입되는지로 확인)
- **SC-007**: 모든 툴바 버튼이 키보드만으로 접근·실행 가능하며 한글 레이블을 노출한다.
- **SC-008**: 확정 시각 디자인과 단일 화면이 시각적으로 일치한다(툴바/에디터/프리뷰/상태바·저장).

## Assumptions

- M1은 단일 문서만 다루며 폴더·다중 문서·대시보드·문서 트리는 후속 마일스톤으로 분리한다.
- 로컬 영속은 브라우저 로컬 저장 메커니즘으로 충족하며, 폴더 기능 도입 시 더 큰 저장소로 이관한다.
  (저장소 기술 선택은 구현 계획 단계에서 결정 — 본 명세는 "재방문 후 보존"이라는 결과만 요구)
- 대상은 데스크톱 사용자 우선. 모바일(에디터/프리뷰 탭 전환)은 본 마일스톤에서 선택 사항이다.
- 에디터/프리뷰 경계 리사이즈, 스크롤 동기화는 권장·선택 항목으로 핵심 흐름의 차단 요소가 아니다.
- UI 텍스트는 한글 기본이며 텍스트 상수 분리로 향후 i18n 여지를 둔다.
- 실제 디스크로의 `.md` 파일 내보내기는 본 마일스톤 범위 밖(선택)이다.
- 시각 디자인은 확정 디자인 레퍼런스를 픽셀 단위로 일치시키되 내부 구조는 복제하지 않는다.
- **배포·운영 제약**: 정적 호스팅 환경에서 서버 의존 없이 빌드·동작해야 한다(계정·백엔드 불필요,
  데이터는 사용자 브라우저에만 잔류). 무설정 정적 빌드 통과가 출시 전제다.
