## 1. 이 글이 다루는 두 문제

저자는 지난 수개월간 두 가지 상호 연결된 문제를 다뤘다고 밝힌다.

| 문제 | 특성 | 평가 난이도 |
|------|------|------------|
| **고품질 프론트엔드 디자인 생성** | 주관적 취향(taste) 영역 | "예쁜가?"는 객관적 답이 없음 |
| **사람 개입 없는 풀 애플리케이션 빌드** | 검증 가능한 정확성·사용성 영역 | 테스트로 부분 검증 가능 |

이 두 영역은 표면적으로 매우 다르지만, 공통적으로 **프롬프트 엔지니어링과 하네스 설계만으로는 천장을 만난다(hit ceilings)**. 저자는 GAN(생성적 적대 신경망)에서 힌트를 얻어 **Generator + Evaluator** 멀티 에이전트 구조를 설계했다.

### 컨텍스트 불안 (Context Anxiety)

- 컨텍스트 윈도우가 채워질수록 모델의 일관성이 떨어진다.
- 일부 모델은 컨텍스트 한계가 다가오면 작업을 조기 마무리하려는 경향을 보인다.
- **컨텍스트 리셋**(=윈도우를 완전히 비우고 새 에이전트 시작 + 구조화된 핸드오프 문서)이 이 문제를 푼다.
- **컴팩션(compaction)과의 차이**: 컴팩션은 같은 에이전트가 압축된 히스토리로 계속 가는 방식. 연속성은 유지하지만 clean slate가 아니라 컨텍스트 불안이 잔존한다.
- 저자에 따르면 Sonnet 4.5는 컨텍스트 불안이 강해 컴팩션만으로 부족했고, 컨텍스트 리셋이 필수였다. 다만 **Opus 4.5에서는 이 경향이 거의 사라져 리셋을 빼도 됐다**.

### 자기 평가 불능 (Self-Evaluation Blindspot)

- 에이전트에게 자기 작업을 평가시키면 자기 결과물을 과도하게 칭찬한다.
- 객관적 검증 기준이 없는 디자인 등 주관 영역에서 더 심각하다.
- 검증 가능한 영역(소프트웨어 테스트)에서도 자기 판단이 흐려져 작업 진행을 방해한다.
- **해결책 — 작업자/평가자 분리**: 평가자를 회의적으로 튜닝하는 것이 생성자를 자기비판적으로 만드는 것보다 훨씬 다루기 쉽다.

## 3. 프론트엔드 디자인: 주관을 측정 가능하게

### 두 가지 통찰

1. **미학은 점수화할 수 없지만, 디자인 원칙을 인코딩한 평가 기준은 만들 수 있다**.  
   "이 디자인이 예쁜가?" → 답하기 어려움  
   "우리 디자인 원칙을 따르는가?" → Claude가 평가할 구체적 기준이 됨
2. **생성과 평가를 분리**하면 생성자를 더 강한 출력으로 끌어가는 피드백 루프가 만들어진다.

### 4가지 평가 기준 (Generator·Evaluator 양쪽 프롬프트에 모두 투입)

| 기준 | 핵심 질문 | 가중치 |
|------|-----------|--------|
| **디자인 품질 (Design Quality)** | 색상·타이포·레이아웃·이미지가 하나의 일관된 분위기·아이덴티티를 만드는가? | ⭐⭐⭐⭐⭐ |
| **독창성 (Originality)** | 의도적 창의 선택이 보이는가? 아니면 라이브러리 기본값·"AI 슬롭(slop)" 패턴(흰 카드 위 보라 그라데이션 등)을 그대로 썼는가? | ⭐⭐⭐⭐⭐ |
| **크래프트 (Craft)** | 타이포 계층, 간격 일관성, 색 조화, 대비비 등 기술적 실행 — competence 체크 | ⭐⭐⭐ |
| **기능성 (Functionality)** | 미학과 무관한 사용성 — 사용자가 인터페이스를 이해하고 주요 액션을 수행할 수 있는가? | ⭐⭐⭐ |

> 💡 **왜 디자인·독창성에 가중치를 더 두는가?**  
> Claude는 크래프트와 기능성은 기본적으로 잘 한다. 문제는 항상 "AI 티 나는" 평범한 결과물에 있다. 가중치 자체가 모델을 미적 위험 감수(aesthetic risk-taking) 방향으로 밀어준다.

### 평가 루프 구성

![02-frontend-eval-loop.svg](https://markflow-r2-uploader.villainscode.workers.dev/images/1778051771953-526e7f50-f0d4-462f-ab83-c31c0ee97a34.svg)

### 흥미로운 관찰들

- **평가 기준 문구가 출력의 성격을 직접 형성**: "the best designs are museum quality" 같은 표현을 추가하니 디자인이 특정 시각적 수렴(visual convergence)으로 끌려갔다. 평가 기준에 사용된 언어 자체가 모델의 결과 캐릭터를 좌우한다.
- **점수가 매번 선형적으로 개선되지는 않음**: 후기 구현이 전반적으로 더 낫지만, 중간 반복본이 더 마음에 드는 경우가 종종 있었다.
- **첫 반복부터 베이스라인보다 눈에 띄게 개선**: 평가자 피드백이 들어가기 전에도 평가 기준 자체가 일반적인 기본값(generic defaults)에서 모델을 멀어지게 했다.

### 기억에 남은 사례 — 가상의 네덜란드 미술관 사이트

- 9번째 반복: 깔끔한 다크 테마 랜딩 페이지 (예측 가능한 수준)
- 10번째 반복: 접근법을 완전히 폐기하고 **공간적 경험**으로 재구상 — CSS perspective로 렌더링한 체커보드 바닥의 3D 룸, 자유 배치된 작품, 클릭/스크롤 대신 문(doorway)으로 갤러리 간 이동
- 단일 패스 생성에서는 본 적 없는 창의적 도약(creative leap)

## 4. 풀스택 코딩으로 확장: 3-에이전트 아키텍처

### 아키텍처 개요


![03-three-agent-architecture.svg](https://markflow-r2-uploader.villainscode.workers.dev/images/1778051797382-87680e49-471a-40f3-b85d-0c032c9985f2.svg)


### 세 에이전트의 역할

#### Planner

- 1~4문장 사용자 프롬프트를 풀 제품 스펙으로 확장한다.
- 야심차게(ambitious about scope) 범위를 잡되, **세부 기술 구현보다 제품 컨텍스트와 고수준 기술 설계에 집중**하도록 프롬프팅한다.
  - 이유: Planner가 세부 기술을 미리 못박으면 그 오류가 하위 구현에 연쇄(cascade)된다.
- AI 기능을 제품 스펙에 자연스럽게 엮을 기회를 찾도록 지시한다.

#### Generator

- 스프린트 단위로 작업, 한 번에 한 기능씩 구현한다.
- 각 스프린트는 **React + Vite + FastAPI + SQLite (이후 PostgreSQL)** 스택으로 앱 구현한다.
- 각 스프린트 끝에서 자체 평가 후 QA(Evaluator)에게 인계.
- git을 사용한 버전 관리.

#### Evaluator

- **Playwright MCP**를 사용해 실제 사용자처럼 동작하는 앱을 클릭해본다.
- UI 기능, API 엔드포인트, DB 상태를 테스트.
- 프론트엔드 실험의 4가지 기준을 풀스택용으로 변형 — 제품 깊이(product depth), 기능성, 비주얼 디자인, 코드 품질.
- 각 기준에 하드 임계값(hard threshold)이 있고, 하나라도 미달하면 스프린트가 FAIL.

### Sprint Contract — 핵심 혁신

**스프린트 시작 전에 Generator와 Evaluator가 "Done의 정의"를 협상한다**:

- 제품 스펙은 의도적으로 고수준이라 user story와 testable implementation 사이의 간극이 있다.
- Generator가 "무엇을 만들 것인가" + "어떻게 검증할 것인가"를 제안.
- Evaluator가 그 제안을 검토, 둘이 합의할 때까지 반복.
- 합의 후 Generator는 그 계약을 기준으로 빌드.
- **통신은 모두 파일 기반**: 한 에이전트가 파일 작성 → 다른 에이전트가 읽고 응답.

이 구조는 스펙에 충실하되 너무 일찍 구현 세부를 못박지 않게 한다.

## 5. 실행 결과와 성능 비교

### 1차 실험 — 2D 레트로 게임 메이커 (Opus 4.5 기반)

> 프롬프트: "Create a 2D retro game maker with features including a level editor, sprite editor, entity behaviors, and a playable test mode."

| 하네스 유형 | 실행 시간 | 비용 |
|-------------|-----------|------|
| Solo (단일 에이전트) | 20분 | $9 |
| Full harness (3-에이전트) | 6시간 | $200 |

**비용 차이는 20배 이상이지만 출력 품질 차이는 즉시 명백했다.**

### Solo 실행 결과 — 깨진 게임


- 레이아웃이 공간을 낭비함 (고정 높이 패널 → 뷰포트 대부분 비어 있음)
- 워크플로우가 경직됨 — UI가 "스프라이트 → 엔티티 → 레벨" 순서를 안내하지 않음
- **결정적으로, 게임이 깨짐**: 엔티티 정의와 게임 런타임 간 와이어링 깨짐. 입력에 반응하지 않음.

### Full Harness 실행 결과 — 작동하는 앱

[스크린샷 영역 — Full Harness 실행 결과의 최초 화면, 스프라이트 에디터, AI 게임 디자인, 게임 플레이 (원문에 5개 이미지 첨부됨)]

- 같은 한 줄 프롬프트가 16-feature × 10-sprint 스펙으로 확장됨
- 스프라이트 애니메이션 시스템, behavior templates, 사운드, **AI 보조 스프라이트 생성기·레벨 디자이너**, 공유 링크 export까지 포함
  
- Planner에게 [Frontend Design Skill](https://github.com/anthropics/claude-code/blob/main/plugins/frontend-design/skills/frontend-design/SKILL.md) 접근권 부여 → 스펙에 비주얼 디자인 언어 작성
- **Play 모드에서 실제로 캐릭터를 움직이고 게임을 할 수 있었음** (Solo 실행은 이게 안 됐다)

### Evaluator가 잡아낸 실제 버그 예시

원문 표를 정리하면 (Sprint 3 한 개에만 27개 기준):

| 계약 기준 | Evaluator 발견 사항 (한글 정리) |
|-----------|---------------------------------|
| 사각형 채우기 도구가 click-drag로 영역 채우기 가능 | **FAIL** — 시작점·끝점에만 타일이 놓이고 영역을 채우지 못함. `fillRectangle` 함수는 존재하나 mouseUp 이벤트에서 호출되지 않는 것이 원인. |
| 배치된 엔티티 스폰 포인트 선택 후 삭제 가능 | **FAIL** — `LevelEditor.tsx:892`의 Delete 키 핸들러가 두 상태 변수를 모두 요구하지만, 엔티티 클릭은 한 쪽만 설정함. 조건을 `selection \|\| (selectedEntityId && activeLayer === 'entity')` 형태로 완화 필요. |
| API로 애니메이션 프레임 순서 변경 가능 | **FAIL** — `PUT /frames/reorder` 라우트가 동적 파라미터 라우트(`/{frame_id}`) 뒤에 등록되어, FastAPI가 `reorder`를 정수형 frame_id로 매칭 시도하다 422 응답을 반환. 라우트 등록 순서 조정 필요. |

### Evaluator 튜닝 비용

- 기본 Claude는 **나쁜 QA 에이전트**다. 합법적 이슈를 찾고도 "큰 문제 아니다"고 결론짓는 패턴 빈발.
- 표면적 테스트만 하고 엣지 케이스를 깊이 파지 않음.
- **튜닝 루프**: Evaluator 로그 읽기 → 내 판단과 다른 사례 찾기 → QA 프롬프트 업데이트 → 반복.
- 여러 라운드의 개발 루프가 필요했다.

## 6. 하네스 단순화 여정 (V1 → V2)

### 단순화의 동기

1. V1 하네스는 부피 크고, 느리고, 비쌌다.
2. **하네스의 모든 컴포넌트는 "모델이 혼자 못 하는 것"에 대한 가정을 인코딩**한다. 이 가정들은 (a) 틀릴 수도 있고 (b) 모델이 좋아지면 빠르게 낡는다.
3. Anthropic의 [Building Effective Agents](https://www.anthropic.com/research/building-effective-agents) 가이드의 원칙: "**가능한 가장 단순한 솔루션을 찾고, 필요할 때만 복잡도를 늘려라**."

### Opus 4.6 출시가 단순화를 가속

Anthropic의 [Opus 4.6 출시 블로그(2026.02.05)](https://www.anthropic.com/news/claude-opus-4-6)는 다음 능력들의 향상을 강조했다:

- 더 신중한 플래닝
- 에이전틱 태스크의 장기 지속
- 큰 코드베이스에서의 안정적 동작
- 코드 리뷰와 디버깅으로 자기 실수를 잡아내는 능력
- 장기 컨텍스트 회수(long-context retrieval) 향상

이것들은 모두 V1 하네스가 보충하려고 만든 능력들이었다. 즉 Opus 4.6에서는 그 보강 컴포넌트들이 점점 불필요해진다.

### V1 → V2 변경 사항

![04-v1-vs-v2-harness.svg](https://markflow-r2-uploader.villainscode.workers.dev/images/1778051859995-c0a6d9b3-1ded-4e94-beb0-1670f41faa96.svg)

### 핵심 결정 — 무엇을 빼고 무엇을 남겼나

| 컴포넌트 | V1 (Opus 4.5) | V2 (Opus 4.6) | 이유 |
|----------|---------------|---------------|------|
| **컨텍스트 리셋** | 필수 | **제거** | Opus 4.5의 컨텍스트 불안이 4.6에서 거의 사라짐 |
| **Sprint 분해** | 필수 | **제거** | 4.6은 큰 작업도 분해 없이 일관되게 처리 |
| **Planner** | 유지 | **유지** | 없으면 Generator가 작업 범위를 과소 잡음(under-scope) |
| **Evaluator (스프린트별)** | 매 스프린트 | **빌드 끝 단일 패스** | 4.6의 솔로 능력 향상으로 매 스프린트 검증 불필요 |
| **자동 컴팩션** | 일부 | **Agent SDK가 자동 처리** | SDK 기본 기능 활용 |

### Evaluator의 역할 재정의

[원문 핵심 — 한글 정리]: **Evaluator는 고정된 yes/no 결정이 아니다. 태스크가 현재 모델의 솔로 능력으로 안정적으로 처리되는 경계를 넘어서 있을 때 비용을 들일 가치가 있다.**

- 4.5에서: 빌드의 대부분이 Generator의 솔로 한계 근처 → Evaluator가 의미 있는 이슈를 광범위하게 잡아냄
- 4.6에서: Generator의 raw 능력이 늘면서 경계가 바깥으로 이동 → 경계 안의 작업은 Evaluator가 불필요한 오버헤드. 경계에 걸친 작업에는 여전히 실질적 가치

### 추가 개선 — 앱 안의 AI 기능

V2에서는 Generator가 앱 자체의 기능을 도구를 통해 구동할 수 있는 적절한 에이전트를 빌드하도록 프롬프트를 개선했다. 이 지식은 비교적 최근 것이라 Claude 학습 데이터에 얇게만 커버되어 있어서 상당한 튜닝이 필요했다.

## 7. V2 하네스 결과 — DAW 빌드 실험

> 프롬프트: "Build a fully featured DAW in the browser using the Web Audio API."

DAW = Digital Audio Workstation (작곡·녹음·믹싱용 음악 제작 프로그램)

### 실행 시간 및 비용 분해

| Agent & Phase | Duration | Cost |
|---------------|----------|------|
| Planner | 4.7분 | $0.46 |
| Build (Round 1) | 2시간 7분 | $71.08 |
| QA (Round 1) | 8.8분 | $3.24 |
| Build (Round 2) | 1시간 2분 | $36.89 |
| QA (Round 2) | 6.8분 | $3.09 |
| Build (Round 3) | 10.9분 | $5.88 |
| QA (Round 3) | 9.6분 | $4.06 |
| **Total V2 Harness** | **3시간 50분** | **$124.70** |

- 대부분의 시간이 빌더에 소비됐고, **스프린트 분해 없이 2시간 이상 일관되게 실행**됐다.

### QA가 잡아낸 실제 갭

**1차 QA 피드백** (요약):
- 디자인 충실도, AI 에이전트, 백엔드는 강함
- 핵심 실패: **Feature Completeness** — 클립을 타임라인에서 드래그/이동 불가, 악기 UI 패널(신스 노브, 드럼 패드) 없음, 시각적 이펙트 에디터(EQ 곡선, 컴프레서 미터) 없음
- 이것은 엣지 케이스가 아니라 DAW를 "사용 가능"하게 만드는 핵심 인터랙션

**2차 QA 피드백** (남은 갭):
- 오디오 녹음이 stub만 있음 (버튼은 토글되나 마이크 캡처 없음)
- 클립 엣지 드래그 리사이즈, 클립 분할 미구현
- 이펙트 시각화가 숫자 슬라이더뿐 (EQ 곡선 없음)

### 한계 인식

- Generator는 자기에게 맡기면 여전히 디테일을 놓치거나 stub 처리하는 경향이 있음
- QA가 last mile 이슈를 잡아내는 가치가 여전히 있음
- **Claude는 실제로 들을 수 없으므로** QA 피드백 루프가 음악적 취향에는 효과가 떨어짐
- 그러나 최종 앱은 작동하는 arrangement view, 믹서, 트랜스포트를 모두 갖춘 기능적 음악 제작 프로그램의 핵심 조각을 가졌다

## 8. 핵심 교훈

저자가 정리한 5가지 교훈:

### ① 빌드 대상 모델로 직접 실험하라

좋은 관행은 늘 자기가 빌드하는 그 모델로 실험하고, 실제 문제에서 trace를 읽고, 원하는 결과로 성능을 튜닝하는 것이다.

### ② 복잡한 작업은 분해 + 전문 에이전트 적용

복잡한 작업에는 분해(decomposition)와 각 측면에 전문 에이전트를 적용하는 데서 헤드룸이 있다.

### ③ 새 모델이 나오면 하네스를 재검토

이미 부하를 떠받치지 않는 부분을 걷어내고, 이전엔 불가능했던 능력을 위한 새 부분을 추가하라.

### ④ 모델이 좋아져도 흥미로운 하네스 조합의 공간은 줄어들지 않는다

저자의 결론: 흥미로운 하네스 조합의 공간은 모델이 좋아져도 줄어들지 않고 **위치가 이동**한다. AI 엔지니어의 흥미로운 일은 다음의 새로운 조합을 계속 찾는 것이다.

### ⑤ Building Effective Agents의 원칙

가능한 가장 단순한 해법을 찾고, 필요할 때만 복잡도를 늘려라.

#### MCP vs CLI 트레이드오프 — 2026년 시점 가이드

Playwright 공식 README의 명확한 권고:

| 사용 시나리오 | 추천 |
|---------------|------|
| 탐색적 테스트, self-QA 루프, 장기 자율 워크플로우 | **MCP** (지속적 브라우저 컨텍스트 가치 > 토큰 비용) |
| 고처리량 코딩 에이전트, 큰 코드베이스 + 테스트 + 추론을 제한된 컨텍스트에서 | **CLI + Skills** (토큰 효율성) |

#### 4-에이전트 테스트 파이프라인 패턴

원문의 3-에이전트 구조에 영감받아, Playwright 커뮤니티에서 4-에이전트 패턴이 등장:

```
Exploration → Test Case Generation → Automation → Maintenance
```

각 에이전트가 하나의 작업만 담당, 파일 기반 핸드오프, shared memory 없음. **app.context.md + JSDoc**로 도메인 지식을 주입.

## 10. 💡 markflow 프로젝트 적용 가이드

> 강의 대상자(개발자)가 markflow 모노레포 컨텍스트에서 즉시 시도해볼 수 있는 실전 패턴.

### 10.1 가장 먼저 할 일 — 평가 기준 명문화

`harness/evaluation-criteria.md` 파일을 만들어 markflow의 "좋은 코드" 정의를 기록한다.

```markdown
# markflow 하네스 평가 기준

## API 일관성 (MUST PASS, 가중치 ⭐⭐⭐⭐⭐)
- [ ] Fastify 5 라우트 핸들러가 RESTful 컨벤션을 따르는가?
- [ ] Drizzle ORM 스키마와 Zod validator가 일치하는가?
- [ ] 에러 응답 포맷이 통일되어 있는가?

## 타입 안전성 (MUST PASS, 가중치 ⭐⭐⭐⭐⭐)
- [ ] `pnpm typecheck`가 0 에러로 통과하는가?
- [ ] `any` 타입 사용 0건인가? (`grep -rn ": any" packages/ apps/`)
- [ ] Drizzle inferred 타입이 컴포넌트까지 전파되는가?

## 테스트 (SHOULD PASS, 가중치 ⭐⭐⭐)
- [ ] Vitest 3 단위 테스트가 비즈니스 로직을 커버하는가?
- [ ] Playwright E2E가 핵심 사용자 플로우를 검증하는가?

## 코드 품질 (SHOULD PASS, 가중치 ⭐⭐⭐)
- [ ] 함수 50줄 이하인가?
- [ ] CodeMirror 6 확장은 별도 모듈로 분리되었는가?
- [ ] Tailwind v4 클래스가 design token을 거치는가?
```

### 10.2 가장 작게 시작하는 Evaluator 에이전트

`.claude/agents/evaluator.md` 작성 예시:

```yaml
---
name: evaluator
description: >
  markflow 코드 리뷰·QA 전담 에이전트.
  Generator가 완성했다고 보고한 작업을 까다롭게 검증한다.
tools: Read, Bash, Glob, Grep
model: opus
---

당신은 markflow의 시니어 엔지니어 + QA 입니다.
다음 평가 기준으로 검토합니다:
@harness/evaluation-criteria.md

판단 원칙:
1. 코드를 직접 실행해 검증한다 (`pnpm test`, `pnpm typecheck`).
2. stub 함수, `// TODO` 주석은 "완료"가 아니다 → FAIL.
3. 타입 에러 1개라도 있으면 → FAIL.
4. 미세한 스타일 차이는 PASS (핵심 기능에 영향 없으면).

판단 사례 (Few-shot):

### FAIL — 올바른 판정
상황: Generator가 "댓글 API 완료" 보고
실제: `// TODO: 권한 체크 추가` 주석 발견
판정: FAIL — 미완성 stub은 완료가 아니다.

### PASS — 올바른 판정
상황: 버튼 hover 색상이 디자인 스펙과 #2px 차이
판정: PASS — 핵심 기능에 영향 없는 미세한 차이.
```

### 10.3 Sprint Contract 도입 — 즉시 시도 가능한 패턴

새 기능 시작 전 `harness/sprints/<번호>-<기능명>.md` 파일을 만든다.

```markdown
# Sprint 7 — 마크다운 라이브 프리뷰

## Generator 제안 (작성: Generator)
구현 범위:
- CodeMirror 6 → unified/remark 파이프라인 연결
- 우측 패널에 sanitize된 HTML 렌더링
- 300ms debounce

검증 방법:
- Vitest: 마크다운 → HTML 변환 5개 케이스
- Playwright E2E: 좌측 입력 → 우측 우표시 < 500ms
- 보안: XSS 페이로드 차단 확인

## Evaluator 검토 (작성: Evaluator)
- [x] 범위 OK
- [ ] 추가 요청: GFM 테이블·코드 펜스 변환 케이스 포함
- [ ] 추가 요청: 큰 문서(10K 라인) 성능 회귀 테스트

## 합의 (양쪽 서명)
✅ Generator: 추가 요청 반영
✅ Evaluator: 위 기준으로 검증 진행
```

### 10.4 Playwright MCP로 self-QA 루프 만들기

```bash
# markflow 디렉토리에서
claude mcp add playwright npx @playwright/mcp@latest
```

CLAUDE.md에 자가 검증 루틴 추가:

```markdown
## 자가 검증 루틴 (PR 전 필수)

새 기능 구현 후 다음을 수행한다:

1. `pnpm dev`로 markflow 실행 (apps/web on :3000, apps/api on :4000)
2. Playwright MCP로 다음 흐름 실행:
   - 새 노트 생성 → 마크다운 입력 → 저장
   - 사이드바에서 노트 선택 → 편집 → 저장
   - 검색창에 검색어 입력 → 결과 클릭
3. 각 단계의 스크린샷 + 콘솔 에러 보고
4. 위 기준 미달 시 즉시 수정
```
